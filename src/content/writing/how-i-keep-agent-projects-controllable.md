## 为什么 Agent 项目容易失控

AI Agent 很擅长执行具体任务，但它不擅长判断"这件事该不该做"。

我遇到过几次这样的情况：给 Agent 一个明确的任务，它完成后顺手把其他东西也改了。有时候改的是 UI 样式，有时候改的是项目结构，有时候甚至把原本能跑的功能改坏了。

问题不在于 Agent 能力不够，而在于我没有给它足够的边界。没有边界的执行，很容易变成"改了不该改的东西"。

所以我现在的方法是：把项目拆成小阶段，每个阶段都有明确目标、禁止事项、验收标准和 commit 节点。

## 我现在的基本原则

### 不让执行 Agent 判断产品方向

执行 Agent 的职责是写代码，不是决定做什么。如果我发现一个功能需要重新思考方向，我会先自己想清楚，或者用另一个模型帮我梳理思路，然后再给执行 Agent 明确的指令。

### 每轮只做一个小目标

一次让 Agent 实现三个功能，不如分三次，每次只做一个。这样如果出问题，范围很小，回滚也容易。

### 先审计，再修改

在让 Agent 修改代码之前，我会先让它读一遍相关文件，确认它理解当前状态。有时候 Agent 会基于它"以为"的代码结构去修改，结果改错了地方。

### 先 build，再 commit

每次 Agent 执行完一轮后，我会先跑 `npm run build`，确认没有编译错误。如果 build 失败，说明这次执行有问题，需要先修复，再 commit。

### 复杂任务拆成多轮

上下文太长会降低执行质量。如果一个任务涉及多个文件、多个步骤，我会拆成多轮对话，每轮只聚焦一个具体问题。

## 我常用的几类指令

### 审计指令

在修改之前，先让 Agent 审计当前状态。比如：

- "请先读一下 src/content.config.ts，确认当前 schema 结构"
- "请检查 src/pages/topics/[slug].astro 的当前实现"
- "请列出 src/content/projects/ 下的所有文件"

这类指令的目的是让 Agent 在修改之前先理解现状。

### 最小修改指令

明确告诉 Agent 只改什么，不改什么。比如：

- "只修改 src/content/writing/example.md，不要改其他文件"
- "只增加 topics 字段，不要删除 tags 字段"
- "不要改 UI 样式，不要改组件结构"

### 回滚指令

如果 Agent 的执行结果不满意，用 git 回滚。比如：

- "git restore ."
- "git reset --hard <commit-hash>"
- "git diff > backup.patch"

关键是：回滚之前要先确认哪个 commit 是干净的。

### 验收指令

每次执行完后，让 Agent 跑验收流程。比如：

- "npm run build"
- "git status"
- "检查 http://localhost:4321/ 是否正常"

### 交接指令

如果要把项目交接给另一个 Agent，写一份简短的交接文档。比如：

- "这个项目使用 Astro + Tailwind，当前状态是..."
- "上一步做了什么，下一步要做什么"
- "有哪些文件不能动"

## 一个实际例子

有一次我让一个 Agent 修改个人主页的 About 区域。它的任务是"优化文案"，结果它把整个 About 重写了，语气从克制变成了求职包装，还顺手改了导航逻辑和 Hero 定位。

我当时的处理步骤是：

1. 用 `git diff` 检查它改了哪些文件
2. 发现它改了 10 个文件，远超"修改 About"的范围
3. 用 `git restore .` 回滚所有改动
4. 重新给指令，明确说"只改 src/content 下的文件，不改组件，不改样式"
5. 这次它只改了 About 文案，没有动其他东西

这次经历让我意识到：边界指令不是可选项，而是必需品。

## 这个方法的局限

### 仍然需要人工判断

Agent 可以执行，但"这样做对不对"还是需要人来判断。我还没有找到一种方式，能让 Agent 完全自主地判断产品方向。

### 不能完全替代审美和产品判断

有一次 Agent 实现了一个功能，build 通过了，功能也能用，但 UI 很丑。这类问题 Agent 判断不了，还是得人工验收。

### 上下文过长仍然会降低执行质量

即使用了"每轮只做一个小目标"的原则，有时候上下文还是会累积到很长。这时候 Agent 的输出质量会明显下降，需要主动清空上下文，重新开始。

## 后续想继续优化的地方

- **交接文档更结构化** - 目前的交接文档比较随意，有时候新的 Agent 接手时还是要重新理解项目
- **验收标准更自动化** - 目前大部分验收还是手动检查，可以考虑写一些自动化测试
- **prompt 模板更稳定** - 目前每次都要重新组织语言，可以积累一些常用的指令模板
- **回滚点更清晰** - 可以建立一个习惯：每个功能完成前都打一个 tag，方便回滚

这些优化不是为了让 Agent 变得更智能，而是为了让我的工作流更稳定。
